# EggClip Windows 客户端互联剪贴板 Roadmap

## 1. 执行规则

本 Roadmap 将 [`Windows客户端互联剪贴板实现方案.md`](./Windows客户端互联剪贴板实现方案.md) 拆成十三个固定任务。

- 按编号顺序推进；前一任务未完成时不开始后一任务。
- 每轮完成一个完整的 `W2W` 任务，包括实现、自动验证和必要文档同步，然后勾选任务标题和子项。
- 子项是同一任务的验收证据，不拆成多个开发轮次。
- 开发期间不新增 Roadmap 任务。发现阻断缺陷时写入“阻断记录”，修复后关闭。
- 不把 POC 连接改名为正式功能，不降低配对、密钥、重放和剪贴板回环防护。
- 除 W2W-01 已明确的握手升级外，协议线格式如需变化，必须暂停当前任务并先更新实现方案、schema、双端类型和共享测试向量。
- 版本号只在全部自动化和双 Windows 真机回归通过后调整。

## 2. 当前基线

- 桌面端可以生成邀请并作为认证服务端接受 HarmonyOS 配对。
- 现有邀请的 `pairingSecret` 尚未绑定网络握手，服务端也没有向客户端发送 Ed25519 身份证明；这是实现 Windows 互联前必须关闭的安全缺口。
- 桌面端有实验性 WebSocket 主动连接，但没有认证客户端握手。
- 正式认证会话已经支持 `ITEM_LIVE`、同步头、范围补齐、ACK、心跳、去重和密钥轮换。
- mDNS 已发布设备 ID、协议版本、端口和能力，但只提供候选地址，不提供信任。
- 桌面端没有“粘贴邀请并加入另一台电脑”的用户流程。
- 当前 `devices` 数据模型没有清楚区分全局设备身份、空间成员关系和拨号角色。

## 3. 里程碑

| 里程碑 | 目标 | 完成条件 |
| --- | --- | --- |
| M1 握手加固与配对客户端 | 关闭现有认证缺口，Windows 能安全完成客户端握手 | Rust、HarmonyOS 和共享向量通过双向认证 |
| M2 空间落库与正式连接 | 加入空间、保存密钥并建立正式认证会话 | 首次配对后进入 ready |
| M3 自动重连与同步 | 自动重连、双向实时同步和离线补齐 | 重启与断网恢复无需重新邀请 |
| M4 产品化验收 | 完整 UI、自动化和双 Windows 真机回归 | 所有完成定义通过 |

## 4. M1：配对客户端

### [x] W2W-01 把邀请秘密和服务端身份绑定到握手

- [x] invitation payload 升级为 version 2，pairing context 使用 `pairing-invitation:v2:<invitationId>`。
- [x] `HelloPayload` 增加 invitation pairing 专用 `pairingProof`，使用 verifier 对固定客户端 claim 计算 HMAC-SHA-256。
- [x] Rust 服务端验证 pairing proof 后才接受 `CLIENT_HELLO`；旧 v1 邀请和缺失 proof 明确拒绝。
- [x] Rust 服务端发送 role=`server` 的 Ed25519 `AUTH_PROOF`，客户端必须验证后才能接受 `AUTH_OK`。
- [x] HarmonyOS 客户端生成 pairing proof、验证服务端 proof，并保持既有可信记录和空间密钥可升级使用。
- [x] 更新实现方案总文档、schema、协议 README、Rust/ArkTS 类型和共享测试向量。
- [x] 覆盖错误秘密、proof 绑定字段变化、服务端冒充、客户端冒充、重放和旧邀请拒绝。

完成标准：不知道邀请秘密或不持有任一端身份私钥的设备都不能完成配对；Rust 与 HarmonyOS 使用同一组向量通过双向认证。

### [x] W2W-02 实现 Rust 邀请解析和短期加入尝试

- [x] 增加 Rust 邀请公开模型，校验 scheme、长度、版本、类型、UUID、公钥、32 字节秘密、期限和最多五个 IPv4 地址。
- [x] 确认码算法与 HarmonyOS 对相同邀请输出一致。
- [x] Tauri command 只向前端返回发行设备、短指纹、空间短 ID、期限、确认码和脱敏地址。
- [x] 完整邀请和秘密只保存在有 TTL 的 Rust 加入尝试中，取消、失败、超时和成功后清除。
- [x] 增加合法、过期、超长、未知版本、字段错误和无效地址测试。

完成标准：桌面端可以安全解析现有邀请，但用户确认前不会发起网络连接，也不会持久化邀请秘密。

### [x] W2W-03 实现桌面配对客户端握手状态机

- [x] 生成本机 X25519 临时密钥和 `CLIENT_HELLO`。
- [x] 校验 `SERVER_HELLO` 的空间、发行设备 ID、身份公钥、临时公钥和 transcript context。
- [x] 使用 Windows 凭据库中的 Ed25519 本机身份签署客户端 `AUTH_PROOF`。
- [x] 验证服务端 role=`server` 的 Ed25519 `AUTH_PROOF` 与邀请发行身份一致。
- [x] 处理 `AUTH_OK`、`AUTH_ERROR`、握手超时、异常关闭和粗粒度拒绝码。
- [x] 正确构造客户端方向的认证传输会话，使用 c2s 出站和 s2c 入站密钥。
- [x] 用共享测试向量和 Rust 服务端测试完成客户端↔服务端互通。

完成标准：内存测试中，桌面 Rust 客户端可以与现有 Rust 配对服务端完成认证并得到可用会话密钥。

## 5. M2：空间落库与正式连接

### [x] W2W-04 调整空间角色和成员数据模型

- [x] 新增本机空间角色：`owner` 或 `member`；既有本机创建空间迁移为 `owner`。
- [x] 将全局设备身份与 `(spaceId, deviceId)` 空间成员关系分离，或实现等价的复合成员主键。
- [x] 新增可信路由：`acceptOnly` 或 `dialCoordinator`，以及最后成功地址。
- [x] 更新 device、clipboard、sync head repository 的查询键和外键。
- [x] migration 支持旧数据库升级、空数据库创建和重复运行，验证行数、索引和外键。

完成标准：同一本机身份可以安全参与现有空间和加入空间，成员、剪贴板来源及同步头不会串空间。

### [x] W2W-05 实现初次加入空间的原子提交

- [x] `AUTH_OK` 后等待并解密初次 `SPACE_KEY_ROTATED`，未收到密钥时不能进入 ready。
- [x] 把空间密钥保存到 Windows 凭据库，只在数据库保存引用。
- [x] 原子写入空间、owner/member 关系、协调端可信设备、拨号角色和最后成功地址。
- [x] 新加入空间设为当前同步空间；不自动合并其他空间的历史或设备。
- [x] 凭据写入后数据库失败时执行补偿删除，不留下半可信记录。
- [x] 重复邀请、空间冲突、同设备不同公钥和密钥版本异常均明确拒绝。

完成标准：首次认证成功后重启应用，空间密钥、协调端身份和可信路由仍可读取；任一中间失败都不会产生可用的半成品空间。

### [x] W2W-06 建立正式出站 WebSocket 认证会话

- [x] 新增正式主动连接入口，与 `connect_poc_peer` 的状态、事件和设备列表严格分离。
- [x] 把客户端方向会话注册到统一 authenticated session registry。
- [x] 同一 `(spaceId, peerDeviceId)` 只保留一个 active session，并正确关闭被替换连接。
- [x] 认证成功后立即交换 `SYNC_HEADS`，同步结束后进入 ready。
- [x] 连接关闭时更新可信设备状态，不删除信任和路由。
- [x] 增加握手超时、帧上限、心跳、空闲断开和通道关闭测试。

完成标准：不用 POC command，Windows B 可以通过正式认证会话连接 Windows A，并在设备列表显示同一可信设备在线。

## 6. M3：自动重连与同步

### [x] W2W-07 实现桌面可信自动重连

- [x] 启动时只为 `dialCoordinator` 路由创建主动连接任务。
- [x] mDNS 仅匹配已保存的协调端设备 ID；优先当前候选地址，回退到上次成功地址。
- [x] 使用带抖动指数退避，并响应网络变化、系统唤醒和 mDNS 地址更新。
- [x] 可信重连 transcript 携带当前空间密钥版本。
- [x] 同版本不重复下发密钥，版本落后时接收新密钥，版本超前时拒绝连接。
- [x] 禁止并发拨号、重复 active session 和失败状态闪烁。

完成标准：关闭并重新打开 Windows B、切换网络或改变 Windows A 地址后，可以自动恢复可信连接，不需要重新粘贴邀请。

### [x] W2W-08 完成 Windows↔Windows 实时剪贴板同步

- [x] Windows A 本地复制通过现有正式 `ITEM_LIVE` 自动发送给 Windows B。
- [x] Windows B 本地复制通过客户端方向会话自动发送给 Windows A。
- [x] 两个方向收到实时事件后按设置自动写入系统剪贴板。
- [x] 远端写入抑制标记只消费一次，相同文本稍后手动复制仍能识别。
- [x] 本地剪贴板操作不等待网络，网络失败不阻塞复制。
- [x] 同步关闭、自动写入关闭、历史关闭和 256 KiB 上限继续生效。

完成标准：两台 Windows 首次连接后无需点击发送按钮，A→B 和 B→A 都能自动同步，且没有本机回声。

### [x] W2W-09 实现同空间实时事件安全转发

- [x] 增加不保存正文的有界 live-event 去重缓存，键为 `(spaceId, originDeviceId, originSeq)`。
- [x] 首次 `ITEM_LIVE` 才能写入本机并转发，重复事件只 ACK。
- [x] 转发时排除来源会话，并为每个目标会话单独生成 AEAD 帧。
- [x] 本机 origin 回流、摘要冲突、跨空间内容和超限内容被拒绝。
- [x] `ITEM_BATCH` 不触发系统剪贴板写入或实时转发。
- [x] 覆盖 A-B-C 环路、Windows+手机+平板和历史关闭场景。

完成标准：协调端同时连接多个设备时，任何成员的首次实时事件都能送达其他在线成员，且不会形成网络环路或剪贴板回环。

### [ ] W2W-10 完成离线补齐、移除和密钥轮换

- [ ] 客户端方向完整处理 `SYNC_HEADS`、`REQUEST_RANGE`、`ITEM_BATCH` 和 `ITEM_ACK`。
- [ ] 断线期间双方产生的事件在重连后补入历史，当前系统剪贴板保持不变。
- [ ] owner 移除成员后关闭其会话、标记 revoked 并轮换空间密钥。
- [ ] 其他在线成员收到新密钥；离线成员可信重连时补发新版本。
- [ ] member 不能生成该空间邀请或执行全局轮换；成员端提供本地“离开空间”。
- [ ] 旧会话、旧密钥和已撤销设备的可信重连均被拒绝。

完成标准：离线补齐、成员移除和空间密钥版本在 Windows↔Windows 方向与现有 HarmonyOS 方向行为一致。

## 7. M4：产品化与验收

### [ ] W2W-11 完成桌面加入和设备管理界面

- [ ] 设备页提供“添加设备”和“加入另一台电脑”两个清晰入口。
- [ ] 加入对话框完成粘贴邀请、脱敏摘要、确认码核对、候选地址和连接进度。
- [ ] 多地址默认自动尝试，用户可以选择地址；手动 IP/端口只放在高级区域。
- [ ] 区分邀请过期、已使用、网络不可达、身份不匹配、认证失败、密钥保存失败和数据库失败。
- [ ] owner/member 操作按角色显示，Rust command 同时执行权限校验。
- [ ] 新设备使用通用“EggClip 设备 #短指纹”名称并支持重命名；POC 不冒充可信设备。
- [ ] 关闭、取消或成功后界面不再保留完整邀请字符串。

完成标准：普通用户不进入诊断模式即可完成两台 Windows 配对、查看在线状态、理解错误并管理可信关系。

### [ ] W2W-12 补齐自动化和发布检查

- [ ] Rust 覆盖邀请解析、客户端握手、migration、路由、客户端会话、重连、同步、转发和轮换。
- [ ] Svelte 覆盖加入流程、状态恢复、地址选择、角色权限和错误文案。
- [ ] Rust 客户端和服务端消费同一组协议测试向量。
- [ ] 运行 `pnpm check`、`pnpm test`、`pnpm build`。
- [ ] 运行 `cargo fmt -- --check`、`cargo check`、`cargo test`。
- [ ] 发布安全检查确认日志、数据库和前端状态不包含邀请秘密、密钥、正文或完整帧。

完成标准：所有桌面自动化和发布检查通过，没有新增未解释警告或敏感信息输出。

### [ ] W2W-13 完成双 Windows 真机验收和文档收尾

- [ ] 两台 Windows 干净数据完成邀请、确认码和首次配对。
- [ ] A→B、B→A、重复文本、超限文本和同步暂停回归通过。
- [ ] 应用重启、系统唤醒、断网恢复、IP 变化、VPN/TUN 和防火墙场景回归通过。
- [ ] 离线补齐只更新历史，不覆盖当前剪贴板。
- [ ] A 同时连接另一台 Windows、手机和平板时实时分发无重复、无串空间、无回环。
- [ ] 成员移除、旧密钥拒绝和剩余设备继续同步回归通过。
- [ ] 更新 `docs/MANUAL_REGRESSION.md`、`README.md`、`desktop/README.md` 和 `DESKTOP_DEVELOPMENT_TODO.md`。

完成标准：两台 Windows 真机和混合设备场景全部通过，阻断缺陷为 0，用户文档与实际操作一致。

## 9. 完成定义

以下条件必须同时满足：

- W2W-01 至 W2W-13 全部勾选。
- invitation v2 秘密证明和双向 Ed25519 身份认证已在 Rust 与 HarmonyOS 上启用。
- Windows A 与 Windows B 可以安全配对、自动重连、双向实时同步和离线补齐。
- 同空间混合设备实时分发没有网络环路和剪贴板回环。
- POC 仍是诊断功能，不具备可信状态。
- 移除设备会关闭会话并轮换空间密钥。
- 自动化、发布安全检查和双 Windows 真机回归全部通过。
- README、桌面 TODO 和手动回归清单与最终行为一致。
