# EggClip 双端国际化实现方案

## 1. 目标

EggClip 桌面端和 HarmonyOS 端增加简体中文、英语界面，并支持跟随系统语言。

本方案解决四类问题：

1. 页面、组件、托盘和系统入口的静态文案。
2. 连接、同步、配对、历史和安全操作的动态状态文案。
3. 日期、时间、数量和英文单复数格式。
4. 数据库中由程序生成的中文空间名、设备名。

国际化不能改变同步协议、安全边界、剪贴板策略或配对数据。语言是本机设置，不在设备之间同步。

## 2. 产品范围

### 2.1 支持的语言

| 设置值 | 界面名称 | BCP 47 标识 | 行为 |
| --- | --- | --- | --- |
| `system` | 跟随系统 / System | 由平台解析 | 中文系统使用简体中文，其他语言回退英语 |
| `zh-CN` | 简体中文 | `zh-CN` | 强制使用简体中文 |
| `en-US` | English | `en-US` | 强制使用英语 |

新安装和升级安装都默认 `system`。旧版本没有语言字段，升级后通过默认值保持兼容。中文系统上的现有用户仍看到中文。

### 2.2 品牌和术语

| 中文 | English | 说明 |
| --- | --- | --- |
| 蛋定 Clip | EggClip | 应用品牌；协议和技术标识始终使用 EggClip |
| 局域网剪贴板同步 | Local clipboard sync | 产品副标题 |
| 同步空间 | Sync space | 一组共享空间密钥和可信成员 |
| 协调端 | Coordinator | 创建并管理空间的设备 |
| 成员端 | Member | 加入已有空间的设备 |
| 可信设备 | Trusted device | 完成身份认证的设备 |
| 配对邀请 | Pairing invitation | 一次性、短期有效的邀请 |
| 确认码 | Confirmation code | 用户核对的短码，不是唯一配对秘密 |
| 自动接收 | Auto-receive | 接收在线实时事件 |
| 自动写入剪贴板 | Copy to clipboard automatically | Windows 收到实时事件后的写入策略 |
| 离线补齐 | Offline catch-up | 重连后只补历史，不覆盖当前剪贴板 |
| 移除并轮换密钥 | Remove and rotate key | 撤销成员并提升空间密钥版本 |
| 离开空间 | Leave space | 成员清除本机空间和信任数据 |

新增或修改用户文案时先复用本表，不为同一概念创造不同英文名称。

### 2.3 不在本阶段实现

- 繁体中文和其他语言。
- 通过同步协议传播语言设置。
- 运行时下载语言包。
- 云端翻译或机器翻译。
- 把用户自定义的设备名、空间名或剪贴板正文翻译成另一种语言。
- 为协议消息增加自然语言错误正文。

## 3. 当前基线

当前代码中约有 54 个 `.svelte`、`.ts`、`.rs`、`.ets` 文件包含中文。部分内容属于测试或内部诊断，但以下用户界面路径必须改造：

- 桌面 Svelte 页面、组件和 `shell.ts` 动态状态。
- Rust command 错误、托盘菜单、在线数量和 tooltip。
- 桌面日期格式中固定的 `zh-CN`。
- HarmonyOS 三个主页面、配对页和设置页。
- HarmonyOS 连接、配对、历史和可信设备 Store。
- HarmonyOS 权限原因、Ability 标签和无障碍文案。
- 数据库中已有的“默认空间”和程序生成设备名称。

HarmonyOS 当前只把应用名称和权限原因等少量文案放入资源文件，大部分界面仍是硬编码中文。桌面端没有语言资源层，两端设置模型都没有语言偏好。

## 4. 核心设计

### 4.1 业务状态与显示文案分离

Store、Service 和 Rust command 不保存最终中文或英文。它们返回稳定状态码和参数：

```ts
export interface UiMessageDescriptor {
  code: UiMessageCode;
  args?: Record<string, string | number>;
}
```

例如：

```ts
{
  code: "trustedDevicesOnline",
  args: { count: 2 }
}
```

页面根据当前语言把它渲染为：

```text
2 台可信设备在线
2 trusted devices online
```

切换语言时，状态码不变，页面重新翻译。不得在连接状态、同步状态、历史状态和配对状态中缓存最终显示文案。

### 4.2 错误边界

面向 UI 的错误使用稳定错误码：

```ts
export interface AppErrorDto {
  code: AppErrorCode;
  args?: Record<string, string | number>;
  safeDetail?: string;
}
```

- `code` 决定用户提示。
- `args` 只携带安全、必要的数字或短标识。
- `safeDetail` 仅用于诊断，不包含剪贴板正文、邀请、密钥、摘要或完整帧。
- 协议仍传输既有错误码，不传中文或英文错误句子。
- 内部日志可以保留技术错误上下文，但不能直接作为最终 UI 文案。

### 4.3 平台资源独立，术语统一

桌面端和 HarmonyOS 端界面结构不同，不共享运行时语言文件。两端共享：

- BCP 47 语言标识。
- 本文的术语表。
- 业务状态码和协议错误码语义。
- 占位参数含义。

两端分别使用适合自身平台的资源机制，并通过自动化检查资源完整性。

## 5. 桌面端实现

### 5.1 Svelte 语言层

新增目录：

```text
desktop/src/lib/i18n/
├─ index.ts
├─ locale.ts
├─ types.ts
├─ format.ts
└─ messages/
   ├─ zh-CN.ts
   └─ en-US.ts
```

职责：

- `types.ts` 定义 `LanguageMode`、`SupportedLocale`、消息 key 和参数类型。
- `locale.ts` 解析设置值与系统语言，提供当前有效语言的 Svelte store。
- `messages/` 保存两种语言的类型化消息。
- `format.ts` 统一日期、时间、数字和单复数格式。
- `index.ts` 导出 `t()`、`formatTime()`、`formatDateTime()` 等公共入口。

中文和英文词典使用同一类型约束。缺少 key、占位参数不一致或使用错误参数时，`pnpm check` 必须失败。

首期不引入大型第三方 i18n 框架。项目只有两种语言，类型化内部词典可以减少依赖，并满足 Svelte 5 的响应式切换需求。如果未来增加三种以上语言或复杂复数规则，再评估 FormatJS、Fluent 或同类方案。

### 5.2 设置模型

桌面 TypeScript 和 Rust `AppSettings` 增加：

```text
languageMode: system | zh-CN | en-US
```

Rust 字段使用 `serde(default)`。设置仍保存在现有 `app_metadata` JSON 中，不新增数据库列。旧 JSON 缺少该字段时回退到 `system`。

设置页“常规”区域增加紧凑的语言选择控件。桌面端切换语言后立即更新：

- 主面板和设置页。
- 当前连接、发送和历史状态。
- 对话框和无障碍标签。
- 原生托盘菜单和 tooltip。

### 5.3 系统语言解析

前端使用 `navigator.languages` 解析 WebView 语言。Rust 托盘使用 Windows 用户区域语言，优先复用现有 `windows-sys` 依赖并增加 Globalization feature。

解析规则：

1. 手动 `zh-CN` 或 `en-US` 直接生效。
2. `system` 下，语言标识以 `zh` 开头时使用 `zh-CN`。
3. 其他语言使用 `en-US`。
4. 无法读取系统语言时使用 `en-US`。

TypeScript 和 Rust 必须对同一测试输入得到相同结果。

### 5.4 日期、时间和数量

删除固定的：

```ts
toLocaleTimeString("zh-CN")
toLocaleString("zh-CN")
```

统一通过 `Intl.DateTimeFormat` 和 `Intl.NumberFormat` 使用当前有效语言。英文数量必须区分单复数，不能通过拼接产生 `1 devices`。

### 5.5 Rust 错误与托盘

新增：

```text
desktop/src-tauri/src/i18n.rs
desktop/src-tauri/src/app_error.rs
```

`i18n.rs` 负责原生界面文案：

- 打开 EggClip。
- 暂停或恢复同步。
- 管理设备。
- 关于。
- 退出。
- 在线可信设备数量。
- 托盘 tooltip。

`TrayStatusState` 保存所有需要更新的 `MenuItem`。保存语言设置后，Rust 立即刷新静态菜单、动态状态和 tooltip。

`app_error.rs` 定义面向 Tauri command 的错误码 DTO。领域模块保留原始错误类型，command 边界将其映射为安全错误码。不得通过前端字符串匹配翻译 Rust 中文错误。

### 5.6 桌面覆盖范围

必须覆盖：

- 主面板、设置、关于页。
- 剪贴板预览和历史列表。
- 同步空间、邀请二维码和加入另一台电脑。
- 可信设备、网络诊断和 POC 诊断。
- 删除、离开、移除和密钥轮换确认框。
- 开机启动、主题和历史设置。
- 托盘菜单、托盘状态和 tooltip。
- `aria-label`、`title`、meta description 和空状态。
- NSIS 内部验收包的中英文系统安装行为，在最终发布阶段单独验证。

## 6. HarmonyOS 实现

### 6.1 使用系统资源

HarmonyOS 使用 ResourceManager 和资源限定目录，不建立第二套 ArkTS 文案字典。

资源结构：

```text
harmony/AppScope/resources/
├─ base/element/string.json
└─ zh_CN/element/string.json

harmony/entry/src/main/resources/
├─ base/element/string.json
└─ zh_CN/element/string.json
```

- `base` 保存英文，作为不支持语言的兜底。
- `zh_CN` 保存简体中文。
- AppScope 覆盖应用名称等全局资源。
- entry 覆盖 Ability、权限、页面、状态和无障碍资源。

静态文案使用 `$r('app.string.key')`。带参数的文案使用 `ResourceManager.getStringSync(resource, ...args)`。不允许在页面中拼接可翻译句子。

参考：

- [HarmonyOS 文档中心](https://developer.huawei.com/consumer/cn/doc/)
- [i18n API](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/js-apis-i18n)
- [ResourceManager API](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/js-apis-resource-manager)

### 6.2 应用偏好语言

新增：

```text
harmony/entry/src/main/ets/services/localization/
├─ LanguagePreferenceService.ets
├─ UiMessageFormatter.ets
└─ UiMessageCodes.ets
```

`LanguagePreferenceService` 使用：

```ts
i18n.System.getAppPreferredLanguage()
i18n.System.setAppPreferredLanguage()
```

映射规则：

| UI 设置 | API 值 |
| --- | --- |
| 跟随系统 | `default` |
| 简体中文 | `zh-CN` |
| English | `en-US` |

系统 API 是 HarmonyOS 语言偏好的事实来源，不在 RDB 中重复保存 `languageMode`。设置页读取系统值并显示选择状态。

SDK 6.1 声明说明应用偏好语言在冷启动后加载资源。因此选择语言后显示本地化提示：“重新打开应用后生效 / Reopen the app to apply the language”。应用不强制退出，也不清除连接、历史或配对数据。

应用处于“跟随系统”时处理配置变化，更新页面语言状态。真机仍需验证当前页面是否自动重建；如果平台只在重新进入页面或冷启动后更新，界面必须给出明确提示。

### 6.3 Store 和 Service 改造

以下模块不再生成最终中文：

- `PairingConnectionStore.ets`
- `PocConnectionStore.ets`
- `PairingStore.ets`
- `HistoryStore.ets`
- `TrustedDeviceStore.ets`
- `SettingsStore.ets`
- 配对、发现、WebSocket、剪贴板和 HUKS Service 的用户错误边界。

它们返回 `UiMessageCode`、领域错误枚举和参数。页面使用 `UiMessageFormatter` 访问资源。

领域 Service 不持有页面 `UIContext`。资源访问只发生在页面或专用本地化适配层，保持现有分层边界。

### 6.4 HarmonyOS 覆盖范围

必须覆盖：

- 首页、最近记录和发送/复制操作。
- 设备页、配对页、扫码和粘贴邀请。
- 自动发现、可信重连和网络排障。
- 设置页、主题、历史、安全自检和恢复操作。
- 悬浮底部页签的无障碍描述。
- Permission reason、Ability label、启动入口和应用描述。
- 手机和平板两套响应式布局。
- PasteButton 周边说明；系统安全组件自身文案遵循系统语言。

## 7. 已有名称数据迁移

用户自定义名称不能翻译。程序生成名称必须随界面语言变化。

### 7.1 数据模型

空间和设备成员增加名称来源：

```text
nameKind = generated | custom
```

- `custom`：显示数据库中的原文。
- `generated`：忽略旧显示名称，按当前语言和短指纹生成界面名称。

如果现有列不允许为空，仍可保留规范化名称，并用 `nameKind` 决定是否翻译。不得把每次语言切换后的翻译结果写回数据库。

### 7.2 migration

桌面 SQLite 和 HarmonyOS RDB 都提供可重复 migration：

1. 新字段默认 `custom`，避免误改用户数据。
2. 只将明确匹配旧版程序生成格式的记录标为 `generated`。
3. 无法确认来源的名称保持 `custom`。
4. 新建默认空间和设备记录时明确写入 `generated`。
5. 用户重命名后改为 `custom`。

升级测试覆盖空数据库、旧数据库、重复 migration 和用户自定义名称保护。

## 8. 文案与安全规则

- 文案 key 使用英文点号命名，如 `pairing.invitation.expired`。
- 同一含义只使用一个 key；不要复制近义句规避参数化。
- 参数占位使用稳定名称，如 `count`、`deviceName`、`expiresAt`。
- 不把剪贴板正文、邀请秘密、密钥、摘要和完整帧放入翻译参数。
- 错误文案说明用户可采取的动作，不展示密码学或数据库内部细节。
- 按钮使用动作动词；标题和说明避免重复。
- 英文按句式大小写，品牌和协议缩写保持既定大小写。
- 测试和内部注释可以使用中文，但用户界面源码中的硬编码文案必须由扫描脚本管控。

## 9. 自动化

新增 `scripts/check-i18n.ps1`，至少检查：

1. 桌面中英文消息 key 完全一致。
2. 两种语言的占位参数一致。
3. HarmonyOS base 和 `zh_CN` string key 完全一致。
4. AppScope 和 entry 必需资源存在。
5. 指定 UI、Store、Service 和托盘文件中没有新增硬编码中文。
6. 没有固定 `toLocaleString("zh-CN")`。
7. 状态 DTO 和 AppError DTO 不包含最终显示文案字段。

该脚本加入 `pnpm release:check` 和仓库发布安全门禁。

测试范围：

- TypeScript：语言解析、key 完整性、参数、日期、数量、即时切换。
- Rust：设置兼容、Windows 语言解析、托盘全部标签、错误码映射。
- ArkTS：语言偏好映射、消息 code 映射、资源 key、旧设置兼容。
- migration：旧程序生成名称、自定义名称、重复执行和回滚边界。
- 页面：中英文长文本、手机/平板、浅色/深色、无障碍标签。

## 10. 手动验收

### 10.1 桌面端

- 中文 Windows 首次启动显示中文。
- 英文 Windows 首次启动显示英文。
- 手动切换立即更新面板、状态、对话框、托盘和 tooltip。
- 重启后保持语言选择。
- “跟随系统”切回系统语言。
- 单台和多台设备的英文数量正确。
- 日期时间使用对应语言格式。
- 配对、断网、认证失败、邀请过期和密钥错误都显示同一语言。
- 英文长文案不造成按钮换行、卡片溢出或局部滚动失效。

### 10.2 HarmonyOS

- 中文和英文系统首次启动选择正确资源。
- 手动选择语言后提示重新打开，冷启动后生效。
- 回到“跟随系统”后使用系统语言。
- 手机和平板首页、设备页、设置页、配对页均无混合语言。
- Permission reason 和 Ability label 使用正确语言。
- 配对、可信重连、PasteButton、历史和 HUKS 自检行为不受影响。
- 英文长文案在手机和平板均不截断关键操作。

### 10.3 跨端

- 两端可以使用不同语言配对和同步。
- 语言设置不进入协议帧，不改变确认码、密钥或设备身份。
- 用户自定义名称保持原文。
- 程序生成名称按本机语言显示。
- 同步历史和系统剪贴板内容不因语言切换发生变化。

## 12. 完成定义

以下条件必须同时满足：

- 两端支持 `system`、`zh-CN`、`en-US`。
- 所有用户界面、托盘、权限原因和无障碍文案都有中英文资源。
- 动态状态和错误使用稳定 code，不缓存最终中文或英文。
- 日期、时间、数字和单复数按当前语言格式化。
- 程序生成名称可本地化，用户自定义名称不变。
- 语言切换不改变协议、安全、配对、历史和剪贴板行为。
- 国际化检查进入发布门禁。
- 桌面、HarmonyOS 手机和平板双语言真机回归通过。
- 文档和应用商店素材与实际行为一致。
